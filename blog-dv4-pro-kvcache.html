DeepSeek-V4-Pro 每个 token 的 KVCache 大小是多少

关键参数

参数值
head_dim512
qk_rope_head_dim64
num_key_value_heads1
num_hidden_layers61
sliding_window128

压缩策略

前 2 层 = 128（HCA），之后交替：偶数索引 = 4（CSA），奇数索引 = 128（HCA）

层数统计：HCA = 31 层，CSA = 30 层

单 KV 对大小（混合精度）

  • RoPE 部分：64 维 × 2 字节（BF16）= 128 字节
  • 非 RoPE 部分：(512 - 64) = 448 维 × 1 字节（FP8）= 448 字节
  • 单个 Key 或 Value = 128 + 448 = 576 字节
  • 一对 KV（MQA，1 组） = 576 × 2 = 1,152 字节

各组件计算

1. SWA（滑动窗口，固定缓存摊销）

SWA 固定缓存 = 61 层 × 128 对 × 1,152 字节 = 8,994,816 字节
SWA 每 token (1M 上下文) = 8,994,816 / 1,048,576 = 9.0 字节

2. HCA（31 层）

HCA 每 token 每层 = 1,152 / 128 = 9 字节
HCA 总计 = 31 × 9 = 279 字节/token

3. CSA（30 层）

CSA 主 KV 每 token 每层 = 1,152 / 4 = 288 字节
索引器键大小 = 64 × 2 + (128 - 64) × 1 = 192 字节
CSA 索引器键每 token 每层 = 192 / 4 = 48 字节
每 CSA 层合计 = 288 + 48 = 336 字节/token
CSA 总计 = 30 × 336 = 10,080 字节/token

汇总

关键技术验证

  • 相比 BF16 GQA8 基线（单层 4,096 字节/token）：V4-Pro 单层平均 ≈ 170 字节 → 24x 压缩率，考虑 FP4 索引器键后可达 31x，接近论文 2% 的声明
  • 相比 DeepSeek-V3.2：V4-Pro 为其 10% → 反推 V3.2 约 101 KB/token

核心公式

单个 KV 对大小 = 2 × (rope_dim × 2 + (head_dim - rope_dim) × 1)
SWA 每 token = (n_layers × sliding_window × kv_pair_size) / seq_len
HCA 每 token = hca_layers × (kv_pair_size / 128)
CSA 每 token = csa_layers × (kv_pair_size / 4 + indexer_key_size / 4)
总计 = SWA + HCA + CSA
