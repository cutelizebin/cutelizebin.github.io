DeepSeek-V4-Pro: KVCache Size Per Token

Compression Strategy

First 2 layers = 128 (HCA), then alternating: even index = 4 (CSA), odd index = 128 (HCA)

Layer count: HCA = 31 layers, CSA = 30 layers

Single KV Pair Size (Mixed Precision)

  • RoPE part: 64 dims × 2 bytes (BF16) = 128 bytes
  • Non-RoPE part: (512 - 64) = 448 dims × 1 byte (FP8) = 448 bytes
  • Single Key or Value = 128 + 448 = 576 bytes
  • One KV pair (MQA, 1 group) = 576 × 2 = 1,152 bytes

Component Calculations

1. SWA (Sliding Window, Fixed Cache Amortization)

SWA fixed cache = 61 layers × 128 pairs × 1,152 bytes = 8,994,816 bytes
SWA per token (1M context) = 8,994,816 / 1,048,576 = 9.0 bytes

2. HCA (31 layers)

HCA per token per layer = 1,152 / 128 = 9 bytes
HCA total = 31 × 9 = 279 bytes/token

3. CSA (30 layers)

CSA main KV per token per layer = 1,152 / 4 = 288 bytes
Indexer key size = 64 × 2 + (128 - 64) × 1 = 192 bytes
CSA indexer key per token per layer = 192 / 4 = 48 bytes
Per CSA layer total = 288 + 48 = 336 bytes/token
CSA total = 30 × 336 = 10,080 bytes/token

Key Validations

  • vs BF16 GQA8 baseline (single layer 4,096 bytes/token): V4-Pro avg ~170 bytes → 24x compression, can reach 31x with FP4 indexer, close to paper's 2% claim
  • vs DeepSeek-V3.2: V4-Pro is 10% → V3.2 ≈ 101 KB/token

Core Formulas

single_kv_pair_size = 2 × (rope_dim × 2 + (head_dim - rope_dim) × 1)
swa_per_token = (n_layers × sliding_window × kv_pair_size) / seq_len
hca_per_token = hca_layers × (kv_pair_size / 128)
csa_per_token = csa_layers × (kv_pair_size / 4 + indexer_key_size / 4)
total = SWA + HCA + CSA
